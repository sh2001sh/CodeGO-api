# CodeGO 任务悬赏设计文档

> 状态：设计提案
> 日期：2026-07-13
> 版本：v0.1
> 适用范围：任务大厅、任务发布、任务接取、材料补充、GitHub 交付、验收结算、争议申诉
> 设计原则：极简发布、按需补充、证据交付、账本可审计

## 1. 文档目标

本功能让用户通过发布和完成 Coding 任务交换平台额度：

- 发布者用普通额度或 Claude 额度悬赏任务。
- 执行者通过 GitHub 仓库、Pull Request 或 Commit 完成交付。
- 发布者验收通过后，悬赏额度结算给执行者。
- 需求不清时，由执行者主动提出材料和问题请求，发布者在平台或 GitHub 回复。
- 产生争议时，平台结合任务原文、沟通记录、GitHub 交付结果和 AI 分析进行人工裁决。

本文档同时作为产品、设计、前端、后端和测试的共同基线。实现时，代码和接口契约优先于本文档中的示例名称。

## 2. 产品摘要

### 2.1 核心价值

任务悬赏不是积分商城，也不是普通的工单系统。它解决的是开发者之间“用可验证的代码交付换取可使用额度”的交易闭环。

核心闭环：

~~~
发布任务
  ↓
冻结发布者额度
  ↓
执行者申请接取
  ↓
发布者确认执行者
  ↓
执行者开发，并按需请求材料
  ↓
GitHub 提交代码、效果图和测试结果
  ↓
发布者验收
  ├─ 通过：额度结算给执行者
  ├─ 要求修改：继续开发
  └─ 拒绝或争议：进入申诉与平台裁决
~~~

### 2.2 最重要的产品决策

| 决策 | 方案 |
|---|---|
| 发布复杂度 | 发布者只填写最少必要字段，不强制填写材料清单、技术环境或详细验收表 |
| 材料补充 | 由执行者在开发过程中手动输入需要的材料和问题 |
| 沟通渠道 | 平台任务讨论为审计记录，GitHub Issue/PR 为开发现场 |
| GitHub 交付 | 必须提交仓库、PR 或 Commit；UI 任务必须附 GitHub 效果图 |
| 悬赏类型 | 每个任务选择普通额度或 Claude 额度，不支持混合悬赏 |
| 额度扣除 | 发布时冻结可用额度，完成时结算，取消或退款时释放 |
| 接取模式 | 默认申请后由发布者确认，不采用无条件抢单 |
| 争议处理 | AI 生成分析建议，平台管理员做最终资金裁决 |

## 3. 设计简报

### 3.1 界面目的

让开发者在几秒内判断一个任务是否值得接取，并让发布者在不填写复杂表单的情况下发布清晰、可执行的 Coding 需求。

### 3.2 目标用户

#### 发布者

- 有明确开发需求但没有足够时间或人力完成。
- 持有普通额度或 Claude 额度。
- 关心任务是否被理解、是否按要求交付、额度是否安全。

#### 执行者

- 熟悉 GitHub 和常见开发流程。
- 希望通过实际 Coding 工作赚取额度。
- 关心需求是否清晰、补充材料是否及时、交付是否有明确证据。

#### 平台管理员

- 处理争议、违规任务、恶意接取和账本异常。
- 需要看到完整任务时间线和资金流水。

### 3.3 使用场景

- 桌面端 1280px 以上：主要开发和验收场景。
- 平板端 768px 以上：浏览任务和查看详情。
- 移动端 320px 以上：查看、申请、回复和提交简单交付，不要求移动端完成复杂编辑。

### 3.4 情绪方向

专业、清晰、可信、克制。界面应像开发者控制台和代码协作工具，不应像游戏悬赏、积分商城或营销活动页面。

### 3.5 视觉记忆点

始终清楚展示两组信息：

1. 任务目前处于什么状态。
2. 悬赏额度是可用、冻结、已结算还是已释放。

## 4. 产品目标与非目标

### 4.1 本期目标

- 建立可公开浏览的 Coding 任务大厅。
- 让发布者用最少字段完成任务发布。
- 让执行者主动提出缺少的需求材料。
- 通过 GitHub 链接和 Commit 固化交付证据。
- 让 UI 任务通过 GitHub 效果图展示最终结果。
- 通过现有 Billing Ledger 完成额度冻结、释放和结算。
- 提供基于任务证据的 AI 争议分析和人工裁决。

### 4.2 非目标

- 本期不做 GitHub 全自动代码托管。
- 本期不要求平台拥有用户仓库写权限。
- 本期不做自动运行用户代码的在线沙箱。
- 本期不做多人协作和复杂里程碑分账。
- 本期不做用户信誉排行榜和复杂等级体系。
- 本期不支持普通额度与 Claude 额度之间的自动转换。

## 5. 术语定义

| 术语 | 定义 |
|---|---|
| 任务 | 发布者提出并设置悬赏的 Coding 需求 |
| 发布者 | 创建任务并提供悬赏额度的用户 |
| 执行者 | 接取任务并提交代码交付结果的用户 |
| 悬赏额度 | 任务完成后发放给执行者的普通额度或 Claude 额度 |
| 冻结 | 发布时从发布者可用余额中暂时扣出，等待结算或释放 |
| 材料请求 | 执行者要求发布者补充需求、截图、权限或环境信息的消息 |
| 交付 | 执行者提交的 GitHub 地址、Commit、效果图、测试结果等证据 |
| 验收 | 发布者根据任务描述和双方后续确认内容判断是否完成 |
| 争议 | 一方对需求理解、交付质量或验收结果提出异议 |

## 6. 角色与权限

### 6.1 发布者权限

- 发布、编辑草稿和取消未开始任务。
- 查看申请者资料和申请说明。
- 确认一名执行者。
- 回复材料请求。
- 查看 GitHub 交付和效果图。
- 通过、要求修改或拒绝交付。
- 在验收期限内发起争议。

### 6.2 执行者权限

- 浏览公开任务。
- 申请接取任务。
- 在开始开发前和开发过程中请求补充材料。
- 查看发布者回复。
- 提交 GitHub 交付结果。
- 响应修改要求。
- 对拒绝结果发起申诉。

### 6.3 管理员权限

- 下架或暂停任务。
- 查看所有任务时间线和账本关联。
- 处理争议并决定全额发放、部分发放或释放额度。
- 处理违规仓库、恶意内容和账号滥用。
- 手工补偿必须通过账本调整并产生审计记录。

## 7. 发布流程

### 7.1 发布者最小表单

发布页面只保留以下必填内容：

| 字段 | 必填 | 设计要求 |
|---|---:|---|
| 任务标题 | 是 | 建议 10–80 个字符，明确动作和目标 |
| 任务描述 | 是 | 支持 Markdown，说明背景、目标和大致结果 |
| GitHub 仓库地址 | 是 | 支持仓库、Issue 或项目地址 |
| 悬赏类型 | 是 | 普通额度 / Claude 额度 |
| 悬赏金额 | 是 | 正整数，使用现有额度单位 |
| 交付截止时间 | 是 | 必须晚于当前时间 |

以下内容全部为可选，不在发布页强制收集：

- 参考效果图。
- API 文档。
- 运行环境。
- 测试账号。
- 品牌素材。
- 详细验收清单。
- 复杂技术约束。

任务描述下方只放一条提示：

> 执行者可以在开发过程中请求补充需求或材料，请尽量在任务讨论中回复。

### 7.2 发布前确认

发布前使用轻量确认面板，不使用复杂的四步向导：

~~~
发布任务

标题：优化控制台首页加载体验
GitHub：github.com/example/project
悬赏：普通额度 10,000
截止：2026-07-30 18:00

发布后将冻结 10,000 普通额度。
任务完成时发放给执行者；取消、过期或申诉退款时释放。

[返回修改] [确认发布并冻结额度]
~~~

### 7.3 发布失败状态

- 额度不足：显示普通额度和 Claude 额度余额，并提供去钱包页入口。
- GitHub 地址无效：定位到地址字段，说明支持的 URL 格式。
- 截止时间无效：提示必须晚于当前时间。
- 重复提交：刷新任务状态，不重复冻结额度。
- 账本服务异常：任务不创建，余额不发生变化。

## 8. 任务大厅设计

### 8.1 页面位置

新增个人导航项：

~~~
个人
├── 钱包
├── 套餐
├── 任务悬赏
├── 拼团大厅
├── 盲盒
└── 邀请
~~~

建议路由：

~~~
/bounties
/bounties/$taskId
~~~

不要复用现有的 /usage-logs/task，避免与已有异步媒体任务概念混淆。

### 8.2 页面布局

任务大厅采用“标题区 + 余额摘要 + Tab + 紧凑列表”的结构，不使用大面积活动 Banner 或重复卡片。

~~~
┌─────────────────────────────────────────────────────────────┐
│ 任务悬赏                                      [发布任务]     │
│ 用 GitHub 交付 Coding 任务，完成后获得普通或 Claude 额度。 │
├─────────────────────────────────────────────────────────────┤
│ 普通额度 12,400       冻结 2,000   Claude 额度 4,800   冻结 0 │
├─────────────────────────────────────────────────────────────┤
│ [任务大厅] [我发布的] [我接取的] [我的申诉]                  │
├─────────────────────────────────────────────────────────────┤
│ [搜索任务或技术栈] [额度类型⌄] [状态⌄] [排序⌄]              │
├─────────────────────────────────────────────────────────────┤
│ 任务标题                    悬赏       截止       状态   操作 │
│ 优化控制台首页加载体验      10,000 普通  7月30日   可申请 查看│
│ 修复 Claude 路由重试逻辑     8,000 Claude  7月26日   可申请 查看│
└─────────────────────────────────────────────────────────────┘
~~~

### 8.3 任务列表项

每一行只承担“判断是否值得查看”的职责：

- 标题。
- 技术标签，最多展示 3 个。
- 悬赏数值和类型。
- 截止时间和剩余时间。
- 发布者名称。
- 当前状态。
- 查看详情或申请接取按钮。

悬赏金额使用等宽数字，额度类型用文字和轻量颜色区分，不使用图标代替文字。

### 8.4 筛选和排序

支持：

- 关键词搜索。
- 普通额度 / Claude 额度。
- 可申请 / 开发中 / 即将截止 / 已完成。
- 技术标签。
- 悬赏范围。
- 按最新发布、额度最高、截止时间排序。

筛选条件写入 URL Search Params，刷新和分享链接后保持状态。

## 9. 任务详情页设计

### 9.1 页面布局

采用“左侧内容、右侧状态与行动”的两栏布局。右侧在桌面端 sticky，在移动端移动到内容顶部。

~~~
┌─────────────────────────────────────────────────────────────┐
│ ← 返回任务大厅                                             │
│ 优化控制台首页加载体验                可申请 · 还剩 12 天    │
│ 发布者：alex · 发布于 7月13日 · React / 性能优化            │
├───────────────────────────────────┬─────────────────────────┤
│ 任务描述                           │ 悬赏                    │
│ 背景与目标                         │ 10,000 普通额度         │
│ GitHub 仓库                        │                         │
│ [打开 GitHub]                      │ 交付截止                │
│                                    │ 2026-07-30 18:00        │
│ 任务讨论                           │                         │
│ [请求补充需求]                     │ 申请接取                │
│                                    │ [申请接取]              │
│ 时间线                             │                         │
│ 发布 → 申请 → 开发 → 交付 → 验收  │                         │
└───────────────────────────────────┴─────────────────────────┘
~~~

### 9.2 详情页内容顺序

1. 标题、状态、悬赏和截止时间。
2. 任务描述。
3. GitHub 仓库入口。
4. 任务讨论和材料请求。
5. 交付记录。
6. 任务时间线。
7. 申诉入口或结算结果。

不要在首屏重复展示长篇规则。规则通过“查看任务规则”折叠展示。

### 9.3 不同身份看到的主按钮

| 用户状态 | 主操作 |
|---|---|
| 未登录 | 登录后申请 |
| 发布者本人 | 管理任务 |
| 普通用户 | 申请接取 |
| 已申请 | 查看申请状态 |
| 当前执行者 | 进入开发 / 提交交付 |
| 已提交 | 等待发布者验收 |
| 已完成 | 查看结算记录 |
| 已争议 | 查看申诉进度 |

## 10. 执行者请求材料设计

### 10.1 交互原则

执行者不需要从固定选项中猜测自己缺什么，直接输入自然语言即可。

按钮文案：

~~~
需要补充需求
~~~

避免使用“发起澄清工单”等偏后台化的词语。

### 10.2 请求面板

~~~
需要发布者补充什么？

请尽量具体，例如：缺少当前页面截图、测试账号、接口返回示例，
或希望确认某个交互应该如何处理。

┌──────────────────────────────────────────────┐
│ 请补充移动端的目标效果图，并确认是否需要适配 │
│ 375px 宽度。                                  │
└──────────────────────────────────────────────┘

☐ 这个问题会阻塞开发

[取消] [发送请求]
~~~

### 10.3 材料请求卡片

任务讨论区中显示：

~~~
执行者 · 7月14日 10:32
需要补充材料 · 阻塞开发

请补充移动端的目标效果图，并确认是否需要适配 375px 宽度。

发布者 · 7月14日 13:05
需要适配移动端，目标效果参考仓库 docs/mobile-reference.png。

[打开 GitHub] [标记为已解决]
~~~

请求状态：

- 待回复。
- 已回复。
- 执行者确认。
- 已关闭。

### 10.4 计时规则

当执行者勾选“阻塞开发”并发送请求时：

- 任务进入 waiting_for_publisher。
- 交付倒计时暂停。
- 发布者回复后进入 publisher_replied。
- 执行者确认后恢复 in_progress。

默认 48 小时没有回复时，执行者可以申请延期或无责取消。平台保留管理员介入能力。

## 11. GitHub 协作设计

### 11.1 MVP 集成方式

MVP 不要求平台取得 GitHub 仓库写权限，只保存以下信息：

- 仓库 URL。
- Issue URL。
- Pull Request URL。
- GitHub 评论 URL。
- Commit SHA。
- 提交时间。

平台可以提供“打开 GitHub”快捷入口，但不强制实现双向评论同步。

### 11.2 GitHub 讨论建议

发布任务时，如果用户提供的是仓库地址，页面提示：

> 建议在对应 GitHub Issue 或 Pull Request 中进行技术讨论，平台会保存相关链接用于验收和申诉。

发布者可以在平台回复，也可以直接在 GitHub 回复。执行者可以在材料请求中补充 GitHub 评论或 Issue 链接。

### 11.3 后续集成

第二阶段可以通过 GitHub OAuth 或 GitHub App 增加：

- Issue/PR 评论同步。
- PR 状态同步。
- Commit 是否存在校验。
- 仓库权限检查。
- GitHub Actions 测试结果同步。

## 12. 交付页面设计

### 12.1 交付表单

执行者点击“提交交付”后，打开全屏页面或宽抽屉：

~~~
提交交付

GitHub 仓库或 PR 地址 *
[https://github.com/example/project/pull/42          ]

最终 Commit SHA *
[a1b2c3d                                             ]

完成说明 *
┌───────────────────────────────────────────────────┐
│ 完成了首页加载优化，并增加了空状态和错误状态。   │
└───────────────────────────────────────────────────┘

效果图（UI 任务必填）
[添加 GitHub 图片地址]

测试结果
┌───────────────────────────────────────────────────┐
│ bun run test 通过                                 │
│ bun run build 通过                                │
└───────────────────────────────────────────────────┘

已知限制
[可选]

[保存草稿] [提交给发布者验收]
~~~

### 12.2 效果图要求

前端、UI、交互类任务必须附带 GitHub 效果图。建议效果图放在：

1. 仓库中的 docs/screenshots/。
2. Pull Request 描述。
3. GitHub Issue 或 PR 的图片附件。

交付说明推荐使用以下结构：

~~~md
## 效果图

### 桌面端
![Desktop](./docs/screenshots/task-123-desktop.png)

### 移动端
![Mobile](./docs/screenshots/task-123-mobile.png)

### 关键状态
![Loading](./docs/screenshots/task-123-loading.png)
![Error](./docs/screenshots/task-123-error.png)
~~~

平台需要：

- 校验图片 URL 格式。
- 展示图片预览。
- 记录图片对应的 Commit SHA。
- 明确图片是否来自最终提交。
- 对失效或无法访问的图片显示错误状态。

非 UI 任务可以将效果图替换为运行截图、测试日志、接口响应或命令行输出。

### 12.3 交付提交后的状态

提交后：

- 任务进入 submitted / reviewing。
- 发布者收到通知。
- 执行者不能静默替换交付地址。
- 如果需要重新提交，必须生成新的 Submission 版本。

## 13. 发布者验收设计

### 13.1 验收面板

发布者进入验收页后看到：

- 原始任务描述。
- 材料请求和回复记录。
- GitHub 仓库、PR 和 Commit。
- 效果图预览。
- 测试结果。
- 变更记录。
- 任务时间线。

如果发布者没有填写详细验收清单，验收依据为：

1. 初始任务描述。
2. 双方在平台或 GitHub 中确认过的补充内容。
3. 最终 GitHub 提交。

### 13.2 验收操作

~~~text
[验收通过并结算]
[要求修改]
[拒绝交付并发起争议]
~~~

要求修改时必须填写一句说明：

~~~text
请补充：移动端 375px 下按钮出现溢出，详情见 GitHub 评论链接。
~~~

建议默认最多 2 轮修改。超过修改次数时，双方可以进入争议流程。

### 13.3 自动结算

发布者提交交付后默认拥有 72 小时验收时间：

- 72 小时内通过：立即结算。
- 72 小时内要求修改：进入修改流程。
- 72 小时内发起争议：冻结额度保持不变，进入人工处理。
- 72 小时内无任何操作：自动结算，避免执行者长期等待。

## 14. 任务状态模型

### 14.1 状态列表

| 状态 | 含义 |
|---|---|
| draft | 草稿，未冻结额度 |
| published | 已发布，等待申请 |
| selecting | 有申请者，等待发布者选择 |
| assigned | 已确认执行者 |
| in_progress | 执行者开发中 |
| waiting_for_publisher | 等待发布者补充材料或回复 |
| publisher_replied | 发布者已回复，等待执行者确认 |
| submitted | 执行者已提交交付 |
| reviewing | 发布者验收中 |
| changes_requested | 发布者要求修改 |
| completed | 已验收并结算 |
| expired | 到期未完成 |
| cancelled | 已取消，额度已释放 |
| disputed | 争议处理中 |
| resolved | 争议已裁决 |

### 14.2 状态转移

~~~text
draft → published
published → selecting
published → cancelled
selecting → assigned
assigned → in_progress
in_progress → waiting_for_publisher
waiting_for_publisher → publisher_replied
publisher_replied → in_progress
in_progress → submitted
submitted → reviewing
reviewing → completed
reviewing → changes_requested
changes_requested → in_progress
reviewing → disputed
disputed → resolved
in_progress → expired
~~~

每一次状态变化必须记录任务事件、操作者和时间。

## 15. 额度账本与结算设计

当前系统已有两类账本账户：

- 普通额度：wallet
- Claude 额度：claude_wallet

任务悬赏必须使用现有 Billing Ledger，不直接修改用户额度结果字段。

### 15.1 发布任务

1. 根据 reward_wallet_type 选择发布者账户。
2. 校验可用余额。
3. 创建任务。
4. 创建 Billing Reservation。
5. 发布者可用余额减少，冻结余额增加。
6. 写入 task_reward_hold 账本记录。

### 15.2 验收结算

在同一数据库事务中：

1. Settlement 消耗发布者 Reservation。
2. Credit 将同等额度记入执行者对应钱包。
3. 写入 task_reward_paid 记录。
4. 写入任务完成事件和通知。

### 15.3 取消和过期

1. 释放发布者 Reservation。
2. 恢复发布者可用余额。
3. 写入 task_reward_release。
4. 更新任务状态。

### 15.4 争议结算

管理员可裁定：

- 全额发放给执行者。
- 部分发放给执行者，剩余额度退回发布者。
- 全额释放给发布者。
- 要求继续修改，暂不结算。

所有结算、释放和发放必须使用幂等键，例如：

~~~text
bounty:{task_id}:reserve
bounty:{task_id}:settle
bounty:{task_id}:credit:{executor_id}
bounty:{task_id}:release
~~~

## 16. 申诉与 AI 分析

### 16.1 申诉入口

申诉入口出现在：

- 发布者拒绝交付后。
- 执行者认为拒绝理由与最终确认需求不一致时。
- 发布者认为执行者没有按最终确认内容完成时。

申诉表单只需要：

- 申诉说明。
- 相关 GitHub 链接。
- 希望的处理结果。
- 可选补充证据。

系统自动附带任务资料，不要求用户重复上传整套材料。

### 16.2 AI 分析输入

AI 分析使用不可变快照：

- 初始任务描述。
- 材料请求和发布者回复。
- GitHub Issue/PR 讨论链接。
- Commit SHA 和交付说明。
- 效果图链接。
- 测试结果。
- 发布者验收意见。
- 修改历史和时间线。

### 16.3 AI 分析输出

~~~text
1. 最终需求摘要
2. 需求是否在过程中发生变化
3. 交付证据是否对应最终 Commit
4. 每个已知要求的通过 / 未通过 / 无法判断
5. 发布者是否及时提供了请求材料
6. 执行者是否按回复后的要求完成
7. 缺失证据
8. 推荐处理结果
9. 置信度和风险提示
~~~

AI 仅提供建议，不能直接操作额度。最终资金裁决由管理员完成。

私有仓库内容必须在获得授权后才能分析；默认优先使用用户提供的文本、Diff、Commit 信息和测试结果。

## 25. 安全与风控

- 所有 GitHub URL 服务端校验，禁止任意协议和脚本地址。
- 不保存用户的 GitHub 密码或长期 Token。
- 私有仓库内容未经授权不得发送给 AI。
- 发布者和执行者不能修改已结算任务的核心记录。
- 任务描述和讨论内容必须进行 XSS/Markdown 安全处理。
- 禁止任务要求提交 API Key、密码或环境变量。
- 对恶意代码、违法内容、钓鱼链接和盗版项目提供举报入口。
- 任务发布、额度冻结、结算、释放和管理员裁决必须写入审计事件。
- 任务接口需要限流，避免批量创建、批量申请和刷通知。

## 26. 关键用户故事与验收标准

### 发布任务

作为发布者，我希望用最少字段发布 Coding 任务，以便快速获得代码交付。

验收标准：

- 只填写标题、描述、GitHub 地址、悬赏和截止时间即可发布。
- 余额不足时不能创建任务。
- 发布成功后悬赏立即进入冻结状态。
- 重复点击发布不会重复创建任务或冻结额度。

### 请求材料

作为执行者，我希望手动说明缺少什么材料，以便发布者准确补充。

验收标准：

- 可以输入任意自然语言请求。
- 可以标记是否阻塞开发。
- 阻塞请求会暂停任务倒计时。
- 发布者可以在平台或 GitHub 回复。
- 执行者确认后请求进入已解决状态。

### GitHub 交付

作为执行者，我希望通过 GitHub 提交代码和效果图，以便发布者能验证最终结果。

验收标准：

- 必须提交 GitHub 地址和 Commit SHA。
- UI 任务没有效果图时不能提交最终交付。
- 平台可以预览可访问的效果图。
- 交付版本和 Commit SHA 固定保存。

### 验收结算

作为发布者，我希望确认完成后立即结算悬赏。

验收标准：

- 验收通过后执行者收到对应额度。
- 发布者 Reservation 被正确结算。
- 结算操作具备幂等性。
- 任务完成、额度变化和账本记录可以互相追溯。

### 争议申诉

作为执行者或发布者，我希望平台能结合完整证据分析争议。

验收标准：

- 申诉自动携带任务描述、材料请求、回复、GitHub 交付和效果图。
- AI 输出结构化分析和推荐结果。
- AI 不能直接发放或退回额度。
- 管理员裁决后额度只结算一次。

## 27. 指标与验收目标

### 核心指标

- 每周成功完成并结算的任务数。
- 任务发布到首次申请的中位时间。
- 申请到确认执行者的中位时间。
- 交付验收通过率。
- 材料请求平均回复时间。
- 任务争议率。
- 自动结算占比。
- 账本对账异常数，目标为 0。

### 产品质量目标

- 发布流程不超过一个页面或一个轻量抽屉。
- 用户可以在 10 秒内看到悬赏类型、金额、状态和截止时间。
- 任务大厅首屏不展示无关营销内容。
- 关键任务操作具备成功、失败、禁用和 loading 状态。
- 亮色和暗色主题均满足正文 4.5:1 对比度。
