# CodeGO 模型分析与月卡续量设计方案

> 状态：设计提案  
> 日期：2026-07-12  
> 原则：不新增一级页面，不增加无关前端负担

## 1. 范围

本次只解决两个问题：

1. 重构现有模型分析页，让用户能比较模型价格、速度、稳定性和真实使用性价比。
2. 为仍在有效期内但额度不足的月卡提供清晰、低门槛的续量渠道。

以下内容不再纳入本期：余额时间线、额度续航预测、复杂套餐推荐系统、独立自定义套餐页面、钱包页新增分析模块。

## 2. 现状判断

### 2.1 模型分析

现有模型分析已经拥有调用量、Token、消费、平均延迟、TPS 和成功率数据，但趋势、占比和排行主要用于描述历史，不能直接帮助用户选择模型。

主要问题：

- 价格和速度分离，用户无法判断是否值得。
- 全站性能与用户自己的实际体验缺少清晰区分。
- 只展示平均值，无法看出慢请求和波动。
- 调用占比与消费排行存在信息重复。
- 不同用途的模型混合排名，结论容易失真。

### 2.2 月卡到期后的继续使用

系统目前已有：

- 月卡续费入口。
- 周卡、日卡等短期补量商品。
- 钱包充值。
- 套餐优先、钱包兜底的扣费顺序。

真正缺失的是针对当前有效月卡直接追加套餐额度的动作。用户的月卡额度提前耗尽时，只能自己判断应该提前续费、购买短期卡还是充值钱包；“短期补量专区”也不等于给当前月卡续量。

## 3. 模型分析页重构

### 3.1 页面位置

直接改造现有 `/dashboard/models`，不新增路由。页面保留原时间范围和模型筛选，主体改为两个视图：

```text
[模型比较] [使用趋势]
```

- 模型比较：新的默认视图，帮助用户选模型。
- 使用趋势：保留现有趋势能力，承接历史分析需求。

调用占比饼图和重复消费排行被替换，不继续叠加新图表。

### 3.2 默认首屏

```text
模型分析
[场景：编程助手⌄] [近7天⌄] [筛选]

┌───────────────────────────────────────────────────────────┐
│  性价比地图                            [地图] [表格]       │
│                                                           │
│  响应快 ↑                                                 │
│         │       ● 模型 A                                  │
│         │  ● 模型 B         ● 模型 C                      │
│         │       ◎ 你常用的模型                            │
│         └────────────────────────────→ 成本更低            │
│                                                           │
│  气泡大小：样本量   颜色：成功率   ◎：用户常用模型         │
└───────────────────────────────────────────────────────────┘

结论：模型 B 比你常用的模型便宜 18%，首 Token 快 120ms，
      最近 7 天成功率相近。                    [查看对比]
```

页面一次只显示地图或表格，避免提高信息密度。

### 3.3 比较指标

| 指标 | 用户含义 | 数据来源 |
|---|---|---|
| 标准任务成本 | 同等输入输出下预计花费 | Billing/Pricing |
| 我的平均成本 | 用户成功请求的实际平均结算 | Ledger + usage evidence |
| 首 Token P50/P95 | 流式请求开始响应的快慢与波动 | Audit projection |
| 输出速度 | 每秒输出 Token | Audit projection |
| 总响应 P50/P95 | 完整请求耗时 | Audit projection |
| 成功率 | 有效成功请求占比 | Audit projection |
| 样本量 | 结论可信程度 | Audit projection |

价格必须由服务端统一计价规则计算，前端不能维护另一份模型价格。

### 3.4 场景分组

只在相同用途内比较模型，避免生成没有意义的总榜：

- 编程助手
- 通用对话
- 长文本
- 推理
- 图像
- 视频

能力标签由管理员维护。模型不满足场景能力门槛时不参与排名。

### 3.5 性价比结论

性价比不是永久固定的模型分数，而是当前场景、时间窗和价格版本下的相对结果。

编程助手场景一期建议权重：

```text
成本 40% + 首 Token 25% + 输出速度 15% + 成功率 20%
```

规则：

- 仅在同一场景内归一化。
- 成功率低于阈值的模型不能标记为“推荐”。
- 样本不足时显示“数据不足”，不强行排名。
- 同时给出一句可解释结论，不只显示综合分。
- 默认突出用户常用模型与最多三个候选模型，不展示几十个气泡。

### 3.6 表格视图

表格用于精确比较，默认列为：

```text
模型 | 标准成本 | 首 Token | 输出速度 | 成功率 | 样本量 | 建议
```

“我的平均成本”和 P95 指标进入展开详情，避免默认表格过宽。移动端只保留模型、成本、首 Token、成功率和建议。

### 3.7 模型详情

点击模型使用现有抽屉展示：

- 与用户当前常用模型的差异。
- 我的数据与全站数据对照。
- 输入、输出、缓存读写价格。
- 近 7 天速度和成功率趋势。
- 复制模型名。
- 创建限定该模型的 API 密钥。

不在本期加入复杂的成本预测、续航预测或套餐联动。

### 3.8 数据补充

现有读模型已有平均延迟、平均 TTFT、TPS 和成功率。本期建议补充：

- TTFT P50/P95。
- 总延迟 P50/P95。
- 模型有效样本量。
- 用户维度模型性能与实际结算聚合。

新增读模型建议：

```text
readmodel.user_model_value_daily
  day, user_id, model_name, group_name,
  request_count, success_count,
  settled_quota,
  ttft_p50_ms, ttft_p95_ms,
  latency_p50_ms, latency_p95_ms,
  output_tokens, generation_ms
```

## 4. 月卡加油包（动态额度）

加油包是当前月卡的上下文补量动作，不是独立套餐，也不进入钱包或 Claude 专用余额。只要用户拥有一张未过期、状态为生效且已开启加油功能的月卡，即可重复购买；不要求已使用额度达到 80%，也不限制距到期日的天数。

### 4.1 规则与权益

- 用户按需输入补充额度，最低 `$10`，并以 `$10` 为步长递增。
- 支付成功后，额度直接追加到目标月卡的 `amount_total` 和该月卡账本；`amount_used`、模型权限、套餐等级与 `end_time` 不变。
- 新增额度和原月卡同一时间到期；支付期间套餐已到期、失效或被续费改变了到期日时，订单拒绝履约。
- 加油不触发拼团奖励、续费阶梯奖励、升级、邀请首购奖励或盲盒折扣；退款按原加油订单处理，不能回退成普通套餐。
- 每个套餐独立配置开关、单价、最低额度与步长。管理员可停用任意档位，服务端始终以订单创建时重新报价后的套餐配置为准。

### 4.2 默认套餐级定价

| 月卡 | 加油价（元 / $1 额度） | 最低额度 | 步长 |
|---|---:|---:|---:|
| Lite | ¥0.175 | $10 | $10 |
| Standard | ¥0.170 | $10 | $10 |
| Pro | ¥0.160 | $10 | $10 |
| Ultra | ¥0.150 | $10 | $10 |

该价格均低于日卡的常规单价，同时随月卡档位逐级降低。加油只保留原月卡的有效期和权益：当用户预计需要长期、大额使用时，升级或购买高档月卡仍然更合适，避免以低档套餐叠加获得高档月卡的模型权益。

### 4.3 交互路径

套餐页仅在“我的当前套餐”中为可用月卡显示“加油”入口。弹窗展示到期时间、最低额度与步长、套餐专属单价、实时应付金额和支付方式；用户确认后进入既有支付渠道。页面明确说明“额度直接追加到当前月卡，到期时间不变”，避免与续费和短期补量卡混淆。

## 5. 页面负担控制

- 钱包页不修改。
- 套餐页只在月卡状态区增加一个上下文按钮，不增加常驻模块。
- 模型分析的新视图替换旧占比和重复排行，不增加页面长度。
- 续量配置只在用户主动点击时打开抽屉。
- 到期状态只保留最近一张相关月卡，不展示完整历史列表。
- 不在模型分析页加入套餐推荐、余额或续航内容。

## 6. 实施顺序

### Phase 1：续量闭环

1. 完成套餐级动态报价、订单与幂等履约。
2. 在套餐页提供当前月卡的上下文入口与支付确认。
3. 为后台套餐编辑页增加按档位配置入口并完成回归。

### Phase 2：模型比较基础版

1. 使用现有价格、平均 TTFT、TPS、成功率实现第一版比较表。
2. 用模型比较替换占比和重复排行。
3. 增加场景分组和用户常用模型标识。

### Phase 3：模型分析增强

1. 增加用户维度模型聚合。
2. 补充 P50/P95 指标。
3. 上线性价比地图和可解释结论。

## 7. 验收标准

### 续量

- 任何未过期的生效月卡均可在两次操作内进入支付确认。
- 支付成功后目标月卡额度即时、幂等增加。
- 新增额度与原月卡同时到期，原到期时间保持不变。
- 管理员可修改每档月卡的加油开关、单价、最低额度与步长。
- 已到期订阅不能续量。
- 用户能明确区分续量、续费和重新购买。
- 套餐页默认信息密度基本不变。

### 模型分析

- 用户能在 30 秒内比较两个模型的成本、速度和成功率。
- 性价比结论均显示时间窗、样本量和场景。
- 样本不足时不输出误导性排名。
- 新视图替换旧图表，页面默认高度不增加。
- 价格数据与 Billing 当前版本一致。
