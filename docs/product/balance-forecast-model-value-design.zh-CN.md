# CodeGO 模型分析与月卡续量设计方案

> 状态：设计提案  
> 日期：2026-07-12  
> 原则：不新增一级页面，不增加无关前端负担

## 1. 范围

本次只解决两个问题：

1. 重构现有模型分析页，让用户能比较模型价格、速度、稳定性和真实使用性价比。
2. 为仍在有效期内但额度不足的月卡提供清晰、低门槛的续量渠道。

以下内容不再纳入本期：余额时间线、额度续航预测、复杂套餐推荐系统、独立自定义套餐页面、钱包页新增分析模块。

## 2. 现状判断

### 2.1 模型分析

现有模型分析已经拥有调用量、Token、消费、平均延迟、TPS 和成功率数据，但趋势、占比和排行主要用于描述历史，不能直接帮助用户选择模型。

主要问题：

- 价格和速度分离，用户无法判断是否值得。
- 全站性能与用户自己的实际体验缺少清晰区分。
- 只展示平均值，无法看出慢请求和波动。
- 调用占比与消费排行存在信息重复。
- 不同用途的模型混合排名，结论容易失真。

### 2.2 月卡到期后的继续使用

系统目前已有：

- 月卡续费入口。
- 周卡、日卡等短期补量商品。
- 钱包充值。
- 套餐优先、钱包兜底的扣费顺序。

真正缺失的是针对当前有效月卡直接追加套餐额度的动作。用户的月卡额度提前耗尽时，只能自己判断应该提前续费、购买短期卡还是充值钱包；“短期补量专区”也不等于给当前月卡续量。

## 3. 模型分析页重构

### 3.1 页面位置

直接改造现有 `/dashboard/models`，不新增路由。页面保留原时间范围和模型筛选，主体改为两个视图：

```text
[模型比较] [使用趋势]
```

- 模型比较：新的默认视图，帮助用户选模型。
- 使用趋势：保留现有趋势能力，承接历史分析需求。

调用占比饼图和重复消费排行被替换，不继续叠加新图表。

### 3.2 默认首屏

```text
模型分析
[场景：编程助手⌄] [近7天⌄] [筛选]

┌───────────────────────────────────────────────────────────┐
│  性价比地图                            [地图] [表格]       │
│                                                           │
│  响应快 ↑                                                 │
│         │       ● 模型 A                                  │
│         │  ● 模型 B         ● 模型 C                      │
│         │       ◎ 你常用的模型                            │
│         └────────────────────────────→ 成本更低            │
│                                                           │
│  气泡大小：样本量   颜色：成功率   ◎：用户常用模型         │
└───────────────────────────────────────────────────────────┘

结论：模型 B 比你常用的模型便宜 18%，首 Token 快 120ms，
      最近 7 天成功率相近。                    [查看对比]
```

页面一次只显示地图或表格，避免提高信息密度。

### 3.3 比较指标

| 指标 | 用户含义 | 数据来源 |
|---|---|---|
| 标准任务成本 | 同等输入输出下预计花费 | Billing/Pricing |
| 我的平均成本 | 用户成功请求的实际平均结算 | Ledger + usage evidence |
| 首 Token P50/P95 | 流式请求开始响应的快慢与波动 | Audit projection |
| 输出速度 | 每秒输出 Token | Audit projection |
| 总响应 P50/P95 | 完整请求耗时 | Audit projection |
| 成功率 | 有效成功请求占比 | Audit projection |
| 样本量 | 结论可信程度 | Audit projection |

价格必须由服务端统一计价规则计算，前端不能维护另一份模型价格。

### 3.4 场景分组

只在相同用途内比较模型，避免生成没有意义的总榜：

- 编程助手
- 通用对话
- 长文本
- 推理
- 图像
- 视频

能力标签由管理员维护。模型不满足场景能力门槛时不参与排名。

### 3.5 性价比结论

性价比不是永久固定的模型分数，而是当前场景、时间窗和价格版本下的相对结果。

编程助手场景一期建议权重：

```text
成本 40% + 首 Token 25% + 输出速度 15% + 成功率 20%
```

规则：

- 仅在同一场景内归一化。
- 成功率低于阈值的模型不能标记为“推荐”。
- 样本不足时显示“数据不足”，不强行排名。
- 同时给出一句可解释结论，不只显示综合分。
- 默认突出用户常用模型与最多三个候选模型，不展示几十个气泡。

### 3.6 表格视图

表格用于精确比较，默认列为：

```text
模型 | 标准成本 | 首 Token | 输出速度 | 成功率 | 样本量 | 建议
```

“我的平均成本”和 P95 指标进入展开详情，避免默认表格过宽。移动端只保留模型、成本、首 Token、成功率和建议。

### 3.7 模型详情

点击模型使用现有抽屉展示：

- 与用户当前常用模型的差异。
- 我的数据与全站数据对照。
- 输入、输出、缓存读写价格。
- 近 7 天速度和成功率趋势。
- 复制模型名。
- 创建限定该模型的 API 密钥。

不在本期加入复杂的成本预测、续航预测或套餐联动。

### 3.8 数据补充

现有读模型已有平均延迟、平均 TTFT、TPS 和成功率。本期建议补充：

- TTFT P50/P95。
- 总延迟 P50/P95。
- 模型有效样本量。
- 用户维度模型性能与实际结算聚合。

新增读模型建议：

```text
readmodel.user_model_value_daily
  day, user_id, model_name, group_name,
  request_count, success_count,
  settled_quota,
  ttft_p50_ms, ttft_p95_ms,
  latency_p50_ms, latency_p95_ms,
  output_tokens, generation_ms
```

## 4. 月卡续量方案

### 4.1 产品定义

新增“月卡续量”，不是新的月卡，也不是钱包充值。

续量包用于：

- 月卡额度提前耗尽或即将耗尽，但距离到期仍有时间。
- 用户希望自定义追加额度，而不是提前购买一个完整的新月卡。
- 新增额度仍应遵守当前月卡的模型范围、权益组和到期时间。

### 4.2 权益去向

续量额度直接追加到用户选择的当前有效月卡中：

- 增加该订阅实例的总额度与可用额度。
- 不进入普通钱包或 Claude 钱包。
- 不延长月卡到期时间。
- 新增额度与原月卡使用同一个 `expires_at`。
- 沿用原月卡的模型限制、分组权益和扣费优先级。
- Ledger 记录为该 subscription account 的额度发放，保证审计与幂等。

订单记录来源为 `subscription_booster`，并关联目标订阅实例 ID。

月卡已到期后禁止续量。因为续量继承原到期时间，给已到期月卡追加额度会立即失效；此时界面只提供“续费”或“重新购买月卡”。

### 4.3 入口

只在已有位置增加上下文动作，不新建页面。

#### 当前月卡仍有效但额度不足

在现有套餐进度旁显示：

```text
Standard 月卡 · 剩余 $8 / $620 · 还有 12 天到期
[续量] [提前续费]
```

#### 月卡已到期

保留最近一张已到期月卡的紧凑状态：

```text
Standard 月卡已于 7月10日到期
[续费] [重新购买月卡]
```

已到期月卡不展示“续量”。

#### 请求因套餐和钱包均不足而失败

错误提示提供明确动作：

```text
当前月卡额度不足，且月卡仍在有效期内。续量后可立即继续调用。
[为月卡续量] [使用钱包]
```

若月卡已经到期，错误提示改为“月卡已到期”，操作只提供续费或重新购买。

### 4.4 购买抽屉

复用现有套餐购买抽屉的支付方式与订单轮询，不创建新页面：

```text
继续使用

为 Standard 月卡续量
当前到期时间：2026-08-10（续量后不变）

输入续量额度
[$ ______]

续量倍率：0.12
应付金额：续量额度 × 0.12

新增额度将进入当前月卡，并与月卡在同一时间到期。

应付：¥12.00（示例：续量 $100）
[确认支付]
```

用户可以自行输入续量额度，但不能修改倍率、模型池或到期时间。服务端按管理员配置的最小额度、最大额度、输入步长和续量倍率生成报价。

### 4.5 与续费的区别

| 操作 | 适合情况 | 结果 |
|---|---|---|
| 续量 | 月卡未到期但额度不足 | 追加到当前月卡，与原月卡同时到期 |
| 提前续费 | 长期稳定使用 | 新的月卡权益按现有续费规则生效 |
| 重新购买月卡 | 月卡已到期且仍长期使用 | 创建新的月卡周期 |

界面必须清楚写明区别，避免用户误以为续量会延长月卡有效期。

### 4.6 定价和风控

- 管理员后台维护续量开关、倍率、最小额度、最大额度、输入步长和每日购买限制。
- 续量倍率默认值为 `0.12`。
- 报价公式为 `应付金额 = 续量额度 × 续量倍率`；金额币种、舍入精度和最低支付金额沿用现有支付配置。
- 最终金额由服务端报价，客户端不计算。
- 订单保存报价版本、续量倍率快照、额度、目标订阅 ID、原到期时间和支付渠道。
- 支付前、支付回调履约前都必须确认目标订阅仍有效；若支付期间订阅到期，不得发放并进入自动退款或人工处理状态。
- 支付回调通过现有订单履约与 Ledger 幂等追加订阅额度。
- 不允许对已退款、风控冻结或来源异常的订阅使用专属优惠。

### 4.7 后端适配

优先复用 Commerce 与 Billing 现有能力：

```text
POST /api/user/subscription/booster/quote
POST /api/user/subscription/booster/purchase
GET  /api/user/subscription/booster/orders/:id
```

建议新增产品类型：

```text
product_type = subscription_booster
benefit_type = subscription_quota
```

支付成功后的权益发放进入目标月卡对应的 subscription ledger account，`reference_type` 记录为 `subscription_booster_order`，并关联目标订阅 ID。订阅快照中的总额度与剩余额度必须在同一事务或既有一致性流程中更新。

### 4.8 生命周期边界

- 仅允许对 `active` 且 `expires_at > now` 的月卡续量。
- 续量不修改 `expires_at`、重置周期、连续续费次数和续费奖励。
- 如果用户有多张有效月卡，必须明确选择目标月卡，不能由服务端静默决定。
- 已到期、已取消、已退款或被冻结的订阅不可续量。
- 续量退款只撤销本次追加权益；若额度已经使用，按现有退款风控规则处理。

## 5. 页面负担控制

- 钱包页不修改。
- 套餐页只在月卡状态区增加一个上下文按钮，不增加常驻模块。
- 模型分析的新视图替换旧占比和重复排行，不增加页面长度。
- 续量配置只在用户主动点击时打开抽屉。
- 到期状态只保留最近一张相关月卡，不展示完整历史列表。
- 不在模型分析页加入套餐推荐、余额或续航内容。

## 6. 实施顺序

### Phase 1：续量闭环

1. 定义续量商品配置和服务端报价。
2. 复用现有支付订单，增加 `subscription_booster` 产品类型。
3. 支付成功后通过 Ledger 向目标订阅账户追加额度。
4. 在有效月卡低余额和额度不足错误状态增加入口；已到期状态只保留续费。
5. 验证重复回调、退款、到账和扣费兜底。

### Phase 2：模型比较基础版

1. 使用现有价格、平均 TTFT、TPS、成功率实现第一版比较表。
2. 用模型比较替换占比和重复排行。
3. 增加场景分组和用户常用模型标识。

### Phase 3：模型分析增强

1. 增加用户维度模型聚合。
2. 补充 P50/P95 指标。
3. 上线性价比地图和可解释结论。

## 7. 验收标准

### 续量

- 有效月卡额度不足用户最多两次操作进入支付确认。
- 支付成功后目标月卡额度即时、幂等增加。
- 新增额度与原月卡同时到期，原到期时间保持不变。
- 管理员可修改续量倍率，初始默认值为 `0.12`。
- 已到期订阅不能续量。
- 用户能明确区分续量、续费和重新购买。
- 套餐页默认信息密度基本不变。

### 模型分析

- 用户能在 30 秒内比较两个模型的成本、速度和成功率。
- 性价比结论均显示时间窗、样本量和场景。
- 样本不足时不输出误导性排名。
- 新视图替换旧图表，页面默认高度不增加。
- 价格数据与 Billing 当前版本一致。
