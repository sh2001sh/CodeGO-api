# ADR-001 先采用模块化单体，再按边界拆服务

## 状态

Accepted

## 背景

CodeGo 当前处于大规模重构阶段，业务复杂度已经足以要求边界清晰，但团队和现有代码还没有稳定到适合直接全量微服务化。若现在强行切微服务，会先得到分布式复杂度，而不是架构收益。

## 决策

CodeGo v2 先采用模块化单体作为主形态，围绕 bounded context 建立清晰模块边界，并把运行时拆成少量二进制：

- `control-api`
- `gateway-api`
- `workflow-worker`
- `ledger-worker`

仅当某个边界满足独立伸缩、独立部署、独立故障域的明确收益时，再把它外拆为独立服务。

## 后果

正面：

- 保留一致性与开发效率
- 显著降低分布式事务复杂度
- 便于快速收敛账本、工作流、网关链路

负面：

- 不是“技术上最炫”的形态
- 某些极端高并发场景下，仍需要后续精确拆分

## 落地约束

- 模块之间必须通过显式应用服务/内部契约交互
- 禁止 controller 跨多个领域直接拼接逻辑
- 运行时拆分不等于领域边界拆分失败
