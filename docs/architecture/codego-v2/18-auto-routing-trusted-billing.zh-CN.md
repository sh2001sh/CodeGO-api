# 单一 Auto 智能路由与可信计费方案

## 1. 定位与目标

本方案将 `new-api` 定位为策略驱动的可靠 API 中转网关，而不是向用户暴露渠道、分组、SLA 或路由策略的复杂平台。

用户只需照常调用所需模型。系统默认使用唯一的 `auto` 分组，自动以最低有效费率选择健康渠道，在故障时恢复路由，并对每次扣费提供可验证的账本记录。

目标如下：

- 用户无需选择路由档位、渠道、分组或 SLA。
- 所有普通 Token 默认使用 `auto`，并隐藏普通用户的手动分组选项。
- 在不牺牲可用性的前提下优先选择最低有效费率。
- 禁用、冷却或故障的渠道不会因亲和性缓存阻断请求。
- 每笔充值、预占、结算、释放和退款均可追溯、幂等且可对账。
- 平台以内部 SLO 驱动自动降权、熔断与恢复，不对用户暴露额外配置。

## 2. 非目标

以下能力不属于本期范围：

- 不提供 `fast`、`stable`、`economy` 等用户侧路由档位。
- 不增加虚拟模型、请求头路由策略或用户可选 SLA。
- 不将渠道名称、评分、密钥或供应商细节暴露给普通用户。
- 不建设创作工作台或内容生产功能。
- 不在首个版本中引入难以验证的实时模型质量评分。

## 3. 用户体验

### 3.1 请求调用

用户继续调用当前兼容接口并指定模型，例如 `/v1/chat/completions`、`/v1/responses` 或图像接口。Token 自动使用 `auto`；用户无须也不能在正常流程中挑选渠道。

用户可感知的结果仅包括：

- 请求默认使用当前最划算且可用的渠道。
- 渠道异常时，系统在首包前自动恢复到备用渠道。
- 钱包和请求日志展示实际费用、请求 ID、成功/失败状态和退款结果。
- 已发生的请求可查询简洁回执，不展示内部渠道信息。

### 3.2 管理员体验

管理员配置全局阈值、观察健康状态、审计路由决策和处理账务异常。管理员不需要为每个用户维护复杂策略。

## 4. Auto 智能路由

### 4.1 固定决策流程

```text
请求模型
  -> 找到用户可访问且支持该模型的分组
  -> 按用户实际有效费率从低到高排序
  -> 在当前分组内选择健康渠道
  -> 首包前失败则记录故障并尝试下一候选
  -> 成功后更新健康数据与亲和性记录
```

候选渠道必须满足以下硬条件：

1. 渠道已启用，且支持请求模型和请求格式。
2. 用户具备对应分组访问权限。
3. 支持当前请求的流式或非流式能力。
4. 渠道和模型未处于冷却状态。
5. 未达到全局并发、速率或预算限制。
6. 用户账户可以完成当前请求的额度预占。

### 4.2 成本与健康规则

路由采用“成本优先、健康兜底”的固定规则：

- 正常情况下按用户实际有效费率升序选择。
- `healthy` 渠道可作为首选候选。
- `degraded` 渠道仅作为备用候选。
- `cooling` 渠道被直接跳过。
- `disabled` 渠道被直接跳过，并立即清除关联亲和性缓存。
- 当最低费率渠道不可用时，按顺序尝试更高费率的可用渠道。

为避免极低价但不稳定的渠道持续影响成功率，系统可配置全局“备用渠道最大溢价”。该阈值只由管理员维护，不向用户暴露。

### 4.3 渠道级健康状态

健康状态以“渠道 + 模型”为粒度保存到 Redis，所有 `gateway-api` 副本共享状态。现有“分组 + 模型”冷却保留为最后防线。

建议初始阈值：

- 同一渠道同一模型连续 3 次可重试失败，进入 2 分钟冷却。
- 冷却结束后仅允许小流量探测恢复。
- 连续成功后恢复为 `healthy`。
- 近 5 分钟成功率持续低于全局阈值时标记为 `degraded`。

健康状态的主要字段：

```text
channel_id
model
state
consecutive_retryable_failures
cooling_until
success_rate_5m
success_rate_15m
ttft_ewma_ms
last_success_at
last_failure_at
```

### 4.4 亲和性失效与恢复

亲和性用于复用成功渠道，但不得阻断可用性：

1. 命中的渠道被禁用、冷却或不再支持模型时，立即删除该记录并回到 `auto` 重新选路。
2. 亲和性渠道在首包前失败时，删除该记录后尝试备用渠道。
3. 备用渠道成功后，自动写入新的亲和性记录。
4. 亲和性缓存为渠道建立反向索引。管理员禁用渠道时，精确清除该渠道的全部关联缓存。

该规则消除“缓存命中已禁用渠道后直接返回 403”的行为。

### 4.5 路由决策审计

每次请求写入轻量 `route_decision` 审计信息：

- 请求 ID、模型、用户组和最终选中渠道。
- 候选数量、被排除的原因、重试次数和是否发生回退。
- 有效费率和价格快照引用。
- 健康状态变化与亲和性命中状态。

普通用户不查看渠道细节；管理员可按请求 ID 定位完整决策链。

## 5. 可信计费

### 5.1 账本原则

账本是额度变化的唯一事实源。余额快照只用于低延迟查询，必须可以由账本事件重建和校验。

每项账务操作具有唯一幂等键：

| 操作 | 幂等键 |
|---|---|
| 充值入账 | 支付流水号 |
| 请求预占 | 请求 ID |
| 实际结算 | 请求 ID + 用量证据 |
| 释放或退款 | 预占 ID + 原因 |

### 5.2 请求计费生命周期

```text
完成路由和价格计算
  -> 生成不可变价格快照
  -> 原子预占额度
  -> 调用上游
  -> 基于实际用量结算
  -> 释放未使用额度或全额释放
  -> 写入账本、审计事件和余额快照
```

价格快照至少包含：模型价格、实际分组倍率、用户特殊倍率、请求类型、计价版本、预估用量和实际用量。后续修改倍率、汇率或模型价格不得影响历史账单。

### 5.3 异常处理

- 路由失败或上游请求尚未发出：全额释放预占。
- 上游部分成功：按可确认的实际用量结算，剩余额度释放。
- 客户端断连：根据上游实际产生的用量结算，不能一律视为免费。
- 重复回调、重复请求或 worker 重试：账本拒绝重复入账。
- `ledger-worker` 重建余额快照并执行账本一致性校验；不一致时记录告警并允许管理员处理。

### 5.4 额度来源

充值、月卡、赠送、兑换码和活动奖励必须是可独立追踪的额度来源。扣费顺序由后台统一配置，每次结算记录各来源的实际分摊。

用户只看到可用额度、冻结额度、已消费和退款；管理员可查看来源、预占、结算、释放和人工调整的完整链路。

## 6. 内部服务保障

本方案不向用户暴露可选 SLA，而以内部 SLO 作为自动化运维目标：

- 可归因平台或上游错误的请求成功率。
- 流式请求的首 token 延迟。
- 首选渠道失败后的自动回退成功率。
- 渠道故障后的恢复时间。
- 预占、结算、释放的账本闭环率。
- 余额快照与账本的一致率。

当指标恶化时，系统自动降权、熔断、清理亲和性、切换备用渠道并告警管理员。管理员仅维护少量全局阈值：失败次数、冷却时长、低成功率阈值、探测频率、首包前重试时限和备用渠道溢价。

## 7. 模块职责

| 模块 | 职责 |
|---|---|
| `gateway/routing` | 候选过滤、费率排序、健康判断、自动回退 |
| `gateway/runtime` | Redis 健康状态、熔断、亲和性反向索引 |
| `gateway/execution` | 记录首包、成功、错误和实际用量 |
| `billing` | 预占、结算、释放、退款、价格快照与幂等账本 |
| `audit` | 路由决策、请求回执、SLO 指标与审计投影 |
| `control-api` | 全局阈值、健康面板、账本对账与人工处置 |
| `ledger-worker` | 余额快照、对账、异常巡检和审计投影 |

`workflow-worker` 不进入同步路由关键路径；后续可用于异步请求、周期性 SLO 报告和故障处置编排。

## 8. 上线计划

### 阶段 1：可观测性与健康状态

1. 将进程内冷却迁移到 Redis。
2. 增加渠道级健康状态和路由决策日志。
3. 建立管理员健康面板与告警。

### 阶段 2：亲和性自动失效与 Auto 切换

1. 为亲和性缓存增加渠道反向索引。
2. 禁用或故障渠道自动清理相关缓存。
3. 以影子模式记录新路由决策并校验。
4. 切换为“低费率优先 + 健康门槛 + 自动回退”。

### 阶段 3：账本切换

1. 所有请求接入预占、结算、释放流程。
2. 通过双写和对账校验账本与现有余额的一致性。
3. 切换余额读取到账本快照，并保留巡检和回滚能力。

### 阶段 4：运营闭环

1. 上线用户请求回执和管理员账本对账。
2. 持续观察内部 SLO，调整全局阈值。
3. 建立故障复盘和渠道恢复流程。

## 9. 验收标准

- 普通用户不需要新增任何路由或 SLA 选择。
- Token 默认使用 `auto`，请求按低费率优先选择健康渠道。
- 已禁用渠道不会因亲和性缓存返回 403。
- 首包前的可重试故障能够自动回退。
- 每笔额度变化可追溯到唯一的预占、结算、释放或充值记录。
- 失败未产生有效用量的请求自动释放额度。
- 余额快照能够由账本重建并通过一致性校验。
