# 人海计划任务与奖金数学模型设计

## 1. 目标

本方案解决 6 个问题：

1. 任务金额对外展示为总奖池金额，不再默认展示“人均奖励”。
2. 除人数任务外，任务允许重复完成，但要限制最大完成次数。
3. 任务完成后，按贡献分把奖金分配到个人，而不是平均分。
4. 组队活动奖励与投稿活动奖励分开展示、分开审核、分开结算。
5. 管理员后台能直接看到成团情况、有效成员情况、任务完成进度和投稿转化。
6. 邀请链接支持注册即自动入队，降低拉新摩擦。

核心目标不是发更多钱，而是用有限奖池，在活动初期换来尽可能多的**有效用户**。

## 2. 有效用户定义

定义用户 `u` 的有效状态：

- `verified(u) ∈ {0,1}`：已完成可验证身份
- `api_key(u) ∈ {0,1}`：已创建 API Key
- `first_call(u) ∈ {0,1}`：已产生首次有效调用
- `retained_7d(u) ∈ {0,1}`：7 天内仍有活跃
- `paid(u) ∈ {0,1}`：已产生真实付费或首充

活动初期的“有效用户”不建议只看注册数，建议定义为：

`effective_user(u) = 1{ verified(u)=1 and api_key(u)=1 and first_call(u)=1 }`

这是第一层有效。

若要用于更高阶奖励，定义二层有效：

`high_quality_user(u) = 1{ effective_user(u)=1 and (retained_7d(u)=1 or paid(u)=1) }`

原因很直接：

- 只看注册数，刷量太容易。
- 只看付费，初期拉新转化太慢。
- `验证 + 建 Key + 首次有效调用` 是初期最适合的有效门槛。

## 3. 奖池总结构

设活动总预算为：

`B_total = B_team + B_submission + B_reserve`

- `B_team`：组队活动奖池，建议占 `60%`
- `B_submission`：投稿活动奖池，建议占 `30%`
- `B_reserve`：运营机动与风控缓冲，建议占 `10%`

推荐初期比例：

- `B_team = 0.60 * B_total`
- `B_submission = 0.30 * B_total`
- `B_reserve = 0.10 * B_total`

理由：

- 初期增长靠组队活动拉新，所以组队池要更大。
- 投稿活动是质量放大器，不是第一阶段的主拉新器。
- 一定要留风控和补贴缓冲，不然后期会被动。

## 4. 任务模型

每个任务定义为：

`Task_i = (pool_i, cap_i, repeatable_i, metric_i, weight_rule_i, review_rule_i)`

字段含义：

- `pool_i`：该任务总奖池金额
- `cap_i`：最大完成次数
- `repeatable_i`：是否允许重复完成
- `metric_i`：完成判定指标
- `weight_rule_i`：贡献分计算规则
- `review_rule_i`：审核/冻结规则

### 4.1 人数任务

人数任务例子：

- 新增有效成员
- 成团
- 满编

人数任务**不走贡献分摊**，采用固定结算。

原因：

- 规则简单，便于早期传播。
- 用户最容易理解。
- 人数任务本质是“拉新转化任务”，贡献者通常明确。

推荐固定分配：

当新增 1 个有效成员时，任务奖池 `pool_headcount` 分为：

- 邀请人：`40%`
- 新成员本人：`30%`
- 团队公共池：`30%`

即：

- `reward_inviter = 0.4 * pool_headcount`
- `reward_invitee = 0.3 * pool_headcount`
- `reward_team_pool = 0.3 * pool_headcount`

团队公共池在后续成团时再释放给团队成员。

### 4.2 非人数任务

除人数任务外，任务允许重复完成，但必须有上限。

设任务 `i` 的最大完成次数为 `cap_i`，已完成次数为 `n_i`。

如果 `n_i >= cap_i`，则该任务停止继续发奖。

单次完成对应的释放奖池：

`release_i = pool_i / cap_i`

如果要更强刺激初期增长，建议前高后低，而不是平均释放：

`release_i(k) = pool_i * alpha_i(k)`

其中：

- `k` 为第 `k` 次完成
- `sum(alpha_i(k)) = 1`
- `alpha_i(1) > alpha_i(2) > ... > alpha_i(cap_i)`

推荐初期用递减系数：

`alpha_i(k) = (cap_i - k + 1) / sum_{j=1..cap_i}(cap_i - j + 1)`

这会让前几次完成更值钱，利于早期冲量。

## 5. 贡献分分配模型

对非人数任务，第 `k` 次完成任务 `i` 后，释放奖金：

`release_i(k)`

团队内每个参与人 `u` 的贡献分定义为：

`score(u, i, k) = base(u, i, k) * quality(u, i, k) * risk(u, i, k)`

其中：

- `base`：基础贡献
- `quality`：质量系数
- `risk`：风控折减

最终个人奖励：

`reward(u, i, k) = release_i(k) * score(u, i, k) / Σ score(v, i, k)`

### 5.1 组队任务贡献分

适用于：

- 团队调用次数里程碑
- 团队消费里程碑
- 周留存/次月留存

定义：

`base_team(u) = w_call * call_share(u) + w_pay * pay_share(u) + w_ret * retain_share(u)`

其中：

- `call_share(u) = calls_u / calls_team`
- `pay_share(u) = spend_u / spend_team`
- `retain_share(u) = retained_u / retained_team`

建议初期权重：

- `w_call = 0.35`
- `w_pay = 0.45`
- `w_ret = 0.20`

原因：

- 调用代表接入完成
- 付费代表更高质量
- 留存防止纯一次性冲量

### 5.2 投稿任务贡献分

投稿任务适用于：

- 内容投稿
- 项目接入
- 社区共建

定义投稿贡献分：

`base_submit(u) = w_owner * owner(u) + w_impl * impl_share(u) + w_quality * quality_score(u) + w_effect * effect_score(u)`

解释：

- `owner(u)`：主提交人标记，主提交人为 `1`，其余为 `0`
- `impl_share(u)`：实际接入/产出占比，取值 `[0,1]`
- `quality_score(u)`：审核质量分，取值 `[0.6,1.2]`
- `effect_score(u)`：传播/使用效果分，取值 `[0.8,1.3]`

建议权重：

- `w_owner = 0.25`
- `w_impl = 0.35`
- `w_quality = 0.20`
- `w_effect = 0.20`

原因：

- 防止只有提交人拿走全部奖励
- 同时避免“谁都说自己做了”的模糊分配

### 5.3 风控折减

若用户命中风控规则，则：

- 正常：`risk = 1`
- 轻风险：`risk = 0.5`
- 冻结：`risk = 0`

常见触发条件：

- 同设备/同 IP 批量注册
- 注册后极短时间内批量建 Key、批量首调
- 投稿内容重复或高度相似
- 团队成员之间明显存在控制关系

## 6. 推荐任务配置

以下是推荐的首期任务设计。

### 6.1 组队活动任务

#### 任务 A：新增有效成员

- 展示金额：总奖池 `pool_A`
- 类型：人数任务
- 是否重复：是
- 上限：无硬上限，但受总活动预算限制
- 结算：固定分配，不走贡献分

推荐：

- 单个有效成员成本控制在 `8 ~ 15 USD`
- 邀请人 / 新成员 / 团队池 = `40 / 30 / 30`

#### 任务 B：成团

- 条件：有效成员数达到 `3`
- 类型：人数任务
- 是否重复：否
- 上限：每队 1 次

推荐：

- `pool_B = 60 USD`
- 队长 `20 USD`
- 成员均分 `40 USD`

#### 任务 C：团队调用里程碑

- 条件：团队累计调用达到 `1000 / 5000 / 10000 / 30000`
- 类型：非人数任务
- 是否重复：是
- 上限：4 次
- 分配：按贡献分

#### 任务 D：团队消费里程碑

- 条件：团队累计真实消费达到 `100 / 300 / 800 / 2000 USD`
- 类型：非人数任务
- 是否重复：是
- 上限：4 次
- 分配：按贡献分

#### 任务 E：留存任务

- 条件：7 日仍活跃人数达到 `2 / 4 / 6`
- 类型：非人数任务
- 是否重复：是
- 上限：3 次
- 分配：按贡献分

### 6.2 投稿活动任务

#### 任务 P1：内容投稿

- 例子：教程、评测、案例、问题解决记录
- 是否重复：是
- 上限：每队每周期 `5` 次
- 分配：按投稿贡献分

#### 任务 P2：项目接入

- 例子：真实站点、产品功能、对外演示项目
- 是否重复：是
- 上限：每队每周期 `3` 次
- 分配：按投稿贡献分

#### 任务 P3：社区共建

- 例子：社群活动、答疑组织、线下分享、联动合作
- 是否重复：是
- 上限：每队每周期 `2` 次
- 分配：按投稿贡献分

## 7. 初期增长最优策略

如果目标是“初期吸引尽可能多的有效用户”，奖励释放顺序应该是：

1. 先重投 `新增有效成员`
2. 再投 `成团`
3. 再投 `团队首个调用/消费里程碑`
4. 最后放大投稿和排行榜

原因：

- 用户从 0 到注册，最缺的是理由。
- 用户从注册到首调，最缺的是即时反馈。
- 用户从首调到留存，才开始需要团队荣誉和长期任务。

因此初期建议：

- `60%` 奖池用于拉新和成团
- `25%` 用于里程碑和留存
- `15%` 用于投稿活动早期内容冷启动

## 8. 管理员后台设计

现有后端已具备基础统计、团队列表、奖励审核、投稿审核接口。建议后台增加以下字段和视图。

### 8.1 团队列表

每支队伍展示：

- 队伍名称
- 队长
- 邀请码
- 邀请链接
- 状态：组队中 / 已成团 / 已锁定
- 当前成员数
- 有效成员数
- 成团率：`effective_members / min_members`
- 累计调用
- 累计消费
- 投稿通过数
- 总发奖金额

### 8.2 任务视图

管理员按任务查看：

- 任务总奖池
- 已释放金额
- 剩余金额
- 已完成次数
- 最大完成次数
- 单次释放金额
- 涉及团队数
- 涉及用户数

### 8.3 投稿审核视图

除基础审核外，要能看到：

- 投稿类型
- 主提交人
- 协作者
- 审核质量分
- 传播效果分
- 建议贡献分
- 建议奖金分配结果

## 9. 邀请链接注册自动入队设计

### 9.1 链接格式

建议统一使用：

`/sign-up?people_plan_invite={invite_code}`

### 9.2 注册流程

1. 用户打开邀请链接
2. 前端把 `people_plan_invite` 存入本地存储
3. 用户完成注册
4. 注册请求带上 `people_plan_invite_code`
5. 后端注册成功后自动尝试加入对应队伍
6. 若队伍已满/失效，注册仍成功，但提示自动入队失败

### 9.3 风控补充

自动入队不应直接发奖励。

必须等用户达到：

`verified + api_key + first_call`

之后才记为有效成员并触发人数任务奖励。

## 10. 是否需要排行榜

结论：**需要，但不能在首期把现金奖励强绑定排行榜。**

### 10.1 需要排行榜的原因

- 能形成社交证明
- 队长更愿意拉人
- 用户知道“别人已经做到什么程度”
- 有利于把团队活跃度从短期冲量拉到持续对抗

### 10.2 不建议首期把奖金完全按排行榜发放

原因：

- 晚加入的团队会觉得无望，降低参与率
- 容易诱发刷量
- 审核成本会上升
- 排名奖励会把用户注意力从“有效用户”转移到“造数字”

### 10.3 推荐方案

首期采用“双轨制”：

1. **基础任务奖励**：按规则结算，人人可得
2. **排行榜荣誉奖励**：小额、非核心、非唯一奖励

推荐排行榜只发：

- 荣誉标识
- 额外曝光
- 小额加成券
- 特殊徽章

不建议首期就把大额现金池完全给前 3 名。

### 10.4 什么时候加大排行榜力度

当满足以下条件再加重排行榜：

- 活跃队伍数 `>= 50`
- 周新增有效用户 `>= 100`
- 投稿审核流程稳定
- 风控拦截能力稳定

在这之前，排行榜应主要服务于：

- 氛围
- 社交传播
- 队长荣誉

而不是成为主奖励分配器。

## 11. 建议落地顺序

### 第 1 阶段

- 奖励分开展示
- 邀请链接注册自动入队
- 人数任务固定结算
- 非人数任务支持重复完成 + 最大次数

### 第 2 阶段

- 贡献分分配引擎
- 管理员后台任务视图
- 投稿活动协作者与建议分账

### 第 3 阶段

- 榜单系统
- 榜单荣誉奖励
- 更细的质量系数和风控折减

## 12. 最终建议

如果你的目标是尽快扩大真实用户池，这套方案里最重要的不是“奖池有多大”，而是下面三件事：

1. **有效用户定义要硬**：至少到首次有效调用
2. **初期奖励要前置**：让前几次成团和前几次任务完成更值钱
3. **排行榜要轻，不要喧宾夺主**：先服务增长，再服务竞争

按这个模型推进，早期最容易拉起来的是：

- 队长有动力拉人
- 新人有动力完成首调
- 团队有动力继续留存
- 投稿活动不会和组队奖励互相打架
